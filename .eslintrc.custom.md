# Custom ESLint Configuration

This document explains the custom ESLint rules configured for this project to catch code quality issues automatically.

## Configuration Overview

Location: `.eslintrc.json`

The configuration is structured into several categories:

### 1. TypeScript Rules (Strict Type Checking)

#### `@typescript-eslint/no-unused-vars: error`
**Purpose**: Catch unused variables, imports, and function parameters  
**Fix**: Remove unused code or prefix with `_` if intentionally unused  
**Example**:
```typescript
// ❌ Bad
import { IconTrendingUp } from '@tabler/icons-react';

// ✅ Good - either use it or remove it
import { IconTrendingUp } from '@tabler/icons-react';
<IconTrendingUp />

// ✅ Good - intentionally unused with underscore prefix
function handler(_event: Event) { ... }
```

#### `@typescript-eslint/no-explicit-any: error`
**Purpose**: Enforce proper type definitions instead of `any`  
**Fix**: Define specific types or interfaces  
**Example**:
```typescript
// ❌ Bad
function updateCell(value: any) { ... }

// ✅ Good
interface CellValue {
  id: string;
  value: string | number;
}
function updateCell(value: CellValue) { ... }
```

#### `@typescript-eslint/explicit-function-return-type: warn`
**Purpose**: Encourage explicit return type declarations  
**Fix**: Add return type annotations  
**Example**:
```typescript
// ⚠️ Warning
function calculateTotal(items) { ... }

// ✅ Good
function calculateTotal(items: Item[]): number { ... }
```

### 2. React Rules (Prevent Common Bugs)

#### `react/no-unescaped-entities: error`
**Purpose**: Prevent rendering issues with special characters  
**Fix**: Use HTML entities or proper encoding  
**Example**:
```typescript
// ❌ Bad
<Text>Today's Activity</Text>

// ✅ Good
<Text>Today&apos;s Activity</Text>
```

#### `react-hooks/exhaustive-deps: error`
**Purpose**: Ensure all dependencies are declared in hooks  
**Fix**: Add missing dependencies or use useCallback/useMemo  
**Example**:
```typescript
// ❌ Bad
useEffect(() => {
  fetchData(userId);
}, []); // Missing userId dependency

// ✅ Good
useEffect(() => {
  fetchData(userId);
}, [userId]);

// ✅ Good - memoize if needed
const columns = useMemo(() => [...], [dependency]);
```

#### `react/jsx-key: error`
**Purpose**: Ensure list items have unique keys  
**Fix**: Add key prop to list items  
**Example**:
```typescript
// ❌ Bad
items.map(item => <div>{item.name}</div>)

// ✅ Good
items.map(item => <div key={item.id}>{item.name}</div>)
```

### 3. General Code Quality Rules

#### `prefer-const: error`
**Purpose**: Use `const` for variables that are never reassigned  
**Fix**: Change `let` to `const` where appropriate  

#### `no-console: warn`
**Purpose**: Prevent console.log in production code  
**Fix**: Use proper logging or remove debug statements  
**Allowed**: `console.warn()` and `console.error()` are permitted

#### `eqeqeq: error`
**Purpose**: Enforce strict equality (===) instead of loose (==)  
**Fix**: Use `===` and `!==` instead of `==` and `!=`

### 4. Import Rules

#### `import/no-duplicates: error`
**Purpose**: Prevent importing from the same module multiple times  
**Fix**: Combine imports into a single statement

#### `import/order: warn`
**Purpose**: Keep imports organized and sorted  
**Groups**: builtin → external → internal → parent → sibling → index  
**Sorting**: Alphabetical, case-insensitive

## Pre-commit Hook Integration

These rules are enforced via Husky pre-commit hook:

```json
// package.json
{
  "lint-staged": {
    "*.{ts,tsx}": [
      "eslint --fix",
      "prettier --write"
    ]
  }
}
```

**Behavior**: 
- Runs automatically before each commit
- Attempts to auto-fix issues where possible
- Blocks commit if unfixable errors remain

## Running Linter Manually

```bash
# Check all files
npm run lint

# Fix auto-fixable issues
npm run lint -- --fix

# Check specific file
npx eslint src/app/clothing/operations/dashboard/page.tsx

# Check and fix specific file
npx eslint --fix src/app/clothing/operations/dashboard/page.tsx
```

## Disabling Rules (Use Sparingly)

```typescript
// Disable for one line
// eslint-disable-next-line @typescript-eslint/no-explicit-any
const data: any = await response.json();

// Disable for entire file (NOT RECOMMENDED)
/* eslint-disable @typescript-eslint/no-explicit-any */
```

**Best Practice**: Only disable rules when absolutely necessary and document why.

## Migration Strategy

When upgrading from warnings to errors:

1. **Phase 1**: Set rule to `"warn"` to identify issues
2. **Phase 2**: Fix all instances across codebase
3. **Phase 3**: Upgrade rule to `"error"` to prevent new violations
4. **Phase 4**: Add to pre-commit hook to enforce
