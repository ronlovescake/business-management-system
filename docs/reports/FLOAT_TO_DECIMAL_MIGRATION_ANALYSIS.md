## 4. Migration plan (recommended phasing)

1. **Pick one model** with the smallest call-site footprint (use the table above to count).
2. **Add a parallel Decimal column** alongside the Float column; backfill via a one-shot script that reads existing rows.
3. **Update writers first** — make every code path that writes the Float field also write the Decimal field. Validate with integration tests.
4. **Update readers** — switch consumers to the Decimal column. Use `Prisma.Decimal` arithmetic helpers; never coerce to `Number` for math.
5. **Drop the Float column** in a follow-up migration once readers are stable for at least one release.
6. **Repeat** per model. Do not attempt a big-bang migration across all 30+ fields at once.

## 5. Risks

- **Type drift**: TypeScript treats `Prisma.Decimal` and `number` as incompatible; expect cascading type errors at every callsite.
- **Serialization**: `Decimal` does not JSON-serialize natively; API responses need explicit `.toString()` or a custom serializer.
- **Inventory & ledger invariants**: any rounding change can shift balances by ±1 centavo. Double-entry assertions must be re-verified on real seed data after each model migration.
- **Frontend math**: charts and totals computed in the browser also need Decimal-equivalent libraries (e.g. `decimal.js`).
