## A ‚Äî Template Creation Rules

| #   | Logic                                                         | Explanation                                                                                                                                                                                 |
| --- | ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Next Due Date must be on or after the accounting cutover date | `createTemplate` rejects a `nextDueDate` that falls before the system's accounting cutover date. This prevents templates from generating backdated drafts outside the active ledger period. |
| 2   | End Date must be on or after Next Due Date                    | If an end date is provided, it must be ‚â• `nextDueDate`. A template cannot end before its first due date.                                                                                    |

## B ‚Äî Draft Generation Logic

| #   | Logic                                                                          | Explanation                                                                                                                                                                                                   |
| --- | ------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 3   | Drafts are generated by advancing the due date month-by-month via a while loop | `generateDueDrafts` repeatedly calls `addOneMonthUtc(currentDate, dayOfMonth)` until reaching the end date or a generation limit.                                                                             |
| 4   | The original day-of-month is preserved across month boundaries                 | `addOneMonthUtc` always snaps back to the original `dayOfMonth`. For example, a template set to the 31st will land on the 31st (or last day of the month where applicable) every cycle.                       |
| 5   | Templates past their end date are auto-deactivated                             | After generating all drafts up to the end date, the template is automatically set to inactive so it no longer generates future drafts.                                                                        |
| 6   | Duplicate drafts are silently skipped                                          | `generateDueDrafts` wraps each insert in a unique-constraint handler. If a draft for a given template + due date already exists, the insert is skipped without error. This makes draft generation idempotent. |

## C ‚Äî Draft Approval (Posting)

| #   | Logic                                                 | Explanation                                                                                                                                                                                                     |
| --- | ----------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 7   | Only DRAFT status records can be approved             | `approveDraft` checks `record.status !== 'DRAFT'` before proceeding. Attempting to approve an already-posted or skipped draft throws an error immediately.                                                      |
| 8   | Double-posting prevention via journal line check      | Before writing new journal lines, `approveDraft` queries for existing journal lines tied to the draft. If any are found, the approval is aborted ‚Äî preventing duplicate journal entries from retried API calls. |
| 9   | Approval creates exactly two journal lines atomically | One debit line and one credit line are written inside a database transaction. If either fails, neither is committed.                                                                                            |

## D ‚Äî Draft Skipping

| #   | Logic                                                              | Explanation                                                                                                                                              |
| --- | ------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 10  | Skipping a draft marks it SKIPPED without creating journal entries | `skipDraft` updates the draft status to "SKIPPED". No ledger entries are created; the period is recorded as intentionally bypassed rather than unpaid.   |
| 11  | Skipped drafts are permanent                                       | There is no "un-skip" operation ‚Äî a skipped draft remains in SKIPPED status. To post for that period, a manual journal entry must be created separately. |
